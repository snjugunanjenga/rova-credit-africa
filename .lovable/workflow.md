# RovaCredit Africa — Engineering Workflow

**Status:** v1.0
**Owner:** Engineering Lead
**Last updated:** 2026-04-24

---

## 1. Branching model — Trunk + short-lived branches

```text
main  ─────●─────●─────●─────●─────●──── (always deployable)
            \         /     /
   feat/lead-form    feat/admin-roles
```

- `main` is always deployable. All work via PRs.
- Branch names: `feat/<scope>`, `fix/<scope>`, `chore/<scope>`, `docs/<scope>`.
- Max branch life: 5 days. Older branches are rebased or closed.
- No long-lived `develop` branch.

## 2. Pull request rules

- Min 1 reviewer (2 for `infra/`, `migrations/`, `auth/`).
- CI must be green: lint, type-check, unit, integration, Playwright smoke.
- PR description uses the template (`What / Why / How tested / Risk`).
- Squash-merge only.
- Auto-delete branch after merge.

## 3. Conventional Commits

```
feat(leads): add eligibility tier persistence
fix(api): clamp installment count to >=1
chore(deps): bump fastapi to 0.115
docs(adr): add ADR-0008 for read replica
```

Used to generate changelogs and trigger semantic-release for the API.

## 4. Code review checklist

- [ ] Tests cover the change
- [ ] No secrets in code
- [ ] OpenAPI / types regenerated if API contract changed
- [ ] Migration is forward-only and reversible-by-design
- [ ] Logs do not contain PII
- [ ] Feature flag added if user-visible
- [ ] Updated docs (`/.lovable/`, README, runbook)

## 5. CI/CD pipeline

```text
GitHub push
    │
    ▼
GitHub Actions
  ├─ lint (ruff, eslint)
  ├─ type-check (mypy, tsc)
  ├─ test (pytest, vitest)
  ├─ playwright (smoke on PR, full on main)
  └─ build container image  ──► Artifact Registry
                                       │
                                       ▼
                              Cloud Build / Cloud Deploy
                                ├─ deploy → dev
                                ├─ smoke ✓
                                ├─ promote → staging
                                ├─ manual approval
                                └─ promote → prod (canary 10% → 100%)

Frontend: Vercel auto-deploys preview per PR; merge to main → prod.
Edge functions / FastAPI: Cloud Run revisions, traffic splitting.
DB migrations: Alembic runs as a separate Cloud Run Job before API rollout.
```

## 6. Release cadence

- Backend: continuous (per merge), with canary.
- Frontend: continuous via Vercel.
- Database migrations: gated; deployed Tue/Thu mornings only.
- Hotfixes: any time, requires post-mortem within 48h.

## 7. Local development

```bash
# Frontend
bun install
bun run dev

# Backend
uv sync
uv run uvicorn app.main:app --reload

# Local infra
docker compose up -d postgres redis
```

`.env.local` is generated by `make bootstrap` which pulls non-prod secrets from Locker.io for the developer's account. No secrets in git.

## 8. Definition of done

A change is "done" when:
1. Merged to `main` and deployed to prod.
2. Telemetry confirms it works (logs, metrics).
3. Docs updated.
4. Linked Linear ticket closed with screenshots / before-after.

## 9. Incident workflow

See `Runbook.md`. Severities Sev1–Sev4. Sev1/Sev2 require post-mortem.
